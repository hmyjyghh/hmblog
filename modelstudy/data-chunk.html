    # 数据分块

    # 1. 什么是分块? 为什么要做数据分块?

    分块就是把大段文本拆分成更小的片段。

    • 这些更小的片段会被存储在向量数据库中, 方便后续检索, 作为LLM生成答案的基础, 也就是作为LLM的上下文。

    为什么要做数据分块?

    • 分块的主要原因是为了确保我们嵌入的内容尽可能少的噪音, 同时仍然保持语义上的相关性。
    • 这对语言模型的输出是有显著的影响的, 提供精确且最相关的信息会大大提高LLM的准确性。

    总的来说，数据分块是为了：

    • 优化相关性: 分块有助于优化从向量数据库中检索内容的相关性。
    • 减少噪音: 主要是为了确保我们嵌入的内容有尽可能少的噪音, 同时保持语义相关。
    • 改善搜索结果: 通过有效的分块策略, 我们可以确保搜索结果准确地捕捉到用户查询的核心。
    • 管理token限制: 这有助于管理我们在每次请求中可以发送的token数量限制。

    # 2. 影响分块大小的因素有哪些?

    1. embedding 模型的选择
    2. 用户Query
    3. 分块的利用

    检索到一个分块后, 需要考虑如何在具体应用中利用这些结果。比如:

    • LLM的token限制是多少?
    • 我们希望在一次LLM请求中包含多少个分块?
    • 我们是否在使用LLM的链式结构, 也就是一个模型的输出传递给另一个模型?

    # 3. 有哪些数据分块的方法?

    1. 固定大小的分块
    2. 内容感知分块，包括句子分割, 递归分块, 特殊分块。
    3. 特殊分块，eg: MarkDown、LaTex
    • 固定大小的分块
    # 可以使用langchain 里面的 `CharacterTextSplitter` 方法
from langchain.text_splitter import CharacterTextSplitter

# 初始化 CharacterTextSplitter
text_splitter = CharacterTextSplitter(
    separator="\n",  # 指定分隔符（默认是\n\n）
    chunk_size=50,   # 每个分块的最大字符数
    chunk_overlap=5, # 分块间重叠字符数（避免语义断裂）
    length_function=len,  # 计算长度的函数（默认按字符数）
)
    • 递归分块
    from langchain.text_splitter import RecursiveCharacterTextSplitter

# 初始化 RecursiveCharacterTextSplitter
# 分隔符优先级：先按段落（\n\n）、再按换行（\n）、再按句子（.）、最后按字符
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=80,          # 每个分块最大字符数
    chunk_overlap=10,       # 分块间重叠字符数（保留上下文）
    separators=["\n\n", "\n", ". ", " ", ""],  # 自定义分隔符优先级（按语义从大到小）
    length_function=len,
)

# 执行切分
chunks = text_splitter.split_text(sample_text)

# 输出结果
print("\nRecursiveCharacterTextSplitter 切分结果：")
for i, chunk in enumerate(chunks):
    print(f"分块 {i+1}：{chunk}（长度：{len(chunk)}）")
    • 特殊分块

    在LangChain中, 我们可以使用MarkdownTextSplitter和 LatexTexxtSplitter

    # 4. 如何找到理想的分块大小?

    这是一个迭代的过程, 没有一个适用于所有情况的分块大小或分块的技术。

    • 但是有一些步骤或者说方法, 帮助找到或接近理想的分块大小。
    1. 首先, 预处理数据。
    • 也就是说,在确定最佳分块大小之前, 我们需要先预处理数据, 确保数据的质量。
    • 比如, 如果数据是通过网页抓取获得的, 那么像HTML标签或者一些只会增加噪音的特定元素需要被移除
    1. 接下来, 我们需要选择分块大小的范围。一旦数据预处理完成, 下一步就是选择一系列可能的分块大小进行测试。
    • 我们应该考虑内容的性质,比如是短消息还是长篇文档;
    • 以及我们将使用的嵌入模型和它的能力, 比如嵌入的模型是否有token限制? 是多少? 做这些的目标是找到保持上下文和维持准确性之间的平衡。
    • 可以从探索各种分块大小开始, 包括较小的分块(比如128或256个token), 用于捕获更细粒度的语义信息; 以及较大的分块(比如512或1024个token),用于保留更多的上下文。
    1. 然后, 需要评估每个分块大小的性能。
    • 使用一个具有代表性的数据集, 为你想测试的分块大小创建嵌入, 并将它们保存到索引中。
    • 然后运行各种查询, 评估质量, 比较不同分块大小的性能。
    • 这是一个迭代的过程, 需要针对不同的查询测试不同的分块大小, 直到确定对于我们的内容和预期查询, 性能最好的分块小。

    # 5. 处理复杂文档时, 最有效的数字化和分块方法是?

    布局感知分块(Layoutaware chunking) 是处理这类复杂文档的理想方法。

    该技术会保留文档的原始布局结构(如标题、页眉、段落), 而实现布局感知分块需要依赖布局理解模型进行数字化处理。

    布局理解模型的核心功能:

    1. 解析文档布局: 自动分析文档的物理结构
    2. 保留格式提取内容: 允许从文档中提取内容时保持原有排版利阅读格式
    3. 识别关键元素: 可定位以下结构组件:
    • 标题
    • 页眉
    • 子标题
    • 正文文本
    • 表格
    • 图表
    • 列表
    • 页脚
    • 页码
    • 键值对
