    # LLM 文本生成阶段

    • 如何一次为单个请求高效生成文本?
    • 如何逐个标记地，为单个输入生成下一个标记

    预测下一个词阶段的优化

    KV缓存, 是LLM推断中最基本的优化之一, 导致了我们所说的标记生成的两个不同阶段之间的分离。

    1. 首先是我们所说的预填充阶段(Prefill), 即使用所有输入生成第一个标记时发生的情况。
    2. 然后是我们所说的解码阶段(Decode), 即使用缓存的K和V值生成后续标记的地方。

    过去的键值是为该特定输入计算然后存储的K和V值,这样我们就不必再次计算 它们。

    但是加入KV Cache 之后,因为现在我们每次只传递一个标记而不是整个输入,所以在解码步骤中,每个标记的时间显著下降,这就是这种大幅度下降的外观。

    这实际上是优化LLMS文本生成的第一个重要里程碑, 这是这些推理系统真正核心的部分。

    但在生成第一个标记之后,出现了一个非常有趣的优化, 即生成后续标记时, 我们实际上只需要将新标记作为输入提供, 然后我们唯一需要重新引入先前输入信息的时间是在进行这个注意力计算时,具体来说是对这些矩阵K和V,我们需要这些矩阵的值,这些值是为先前的标记计算出来的, 现在计算这个值将帮助我们生成下一个标记。

    因此,你会发现,实际上,先前标记的这些K和V的值可以被缓存, 这样我们就不必在每次调用这个计算之间重新计算它们, 然后只需要进行一个非常轻量级的连接操作, 在这个操作中我们获取缓存的值, 附加新标记的新K和V值,进行注意力计算,重新缓存这些值,然后生成下一个标记。

    这就是KV缓存的整体概念，是LLM推断中最基本的优化之一, 导致了我们所说的标记生成的两个不同阶段之间的分离。

    1. 首先是我们所说的预填充阶段(Prefill), 即使用所有输入生成第一个标记时发生的情况。
    2. 然后是我们所说的解码阶段(Decode), 即使用缓存的K和V值生成后续标记的地方。

    这实际上使我们能够避免为输入序列中的所有过去的标记一遍又一遍地执行所有那些冗余的计算。

    在这之上,还有很多复杂技术, 比如页面侦测是一个非常流行的技术, 它真的试图优化KV缓存的利用, 无论是在内存和计算上还是在CUDA层发生了什么,所以我们可以在此基础上添加更多的优化层,像Lora这样的系统实现了这些优化。

    # 批处理阶段优化

    可以了解到，批处理是如何通过增加吞吐量来降低延迟的？

    • 将多个请求一起批处理， 如何在处理更多请求的同时（称为吞吐量） 和快速响应 任何一个请求(称为延迟)之间进行权衡。
    • 批处理是一种我们可以使用的技术，专们用于提高吞吐量，这在我们想要构建一个能够处理多个并发请求的服务器系统时非常重要，而不是像我们在这里处理的单个请求一样。

    # 动态批处理 VS 连续批处理(Continuous Batching)

    好的，这是一个非常核心且重要的问题。简单来说，连续批处理是动态批处理的一种更高级、更极致的演进形式，它解决了动态批处理在长文本、高并发等场景下的根本性瓶颈。

    我们来详细拆解一下两者的区别。

    # 1. 动态批处理

    动态批处理是相对于静态批处理而言的。

    • 静态批处理：在服务启动或处理前，将请求预先分组，形成一个或多个固定大小的批次。处理过程中，一个批次内的所有请求必须同时开始、同时结束。缺点很明显：如果一批次里有一个请求处理很慢（比如生成长文本），那么整个批次的其他请求即使完成了，也要等待这个慢请求，造成资源浪费（著名的“木桶效应”）。

    • 动态批处理：

      • 核心思想：当一个推理实例（如GPU）空闲时，它不会等待一个固定的批次，而是实时地从请求队列中取出尽可能多的请求，组成一个新批次进行处理。批次大小 batch_size 不再是固定的，而是根据当前队列长度动态变化的。
      • 工作流程：
        1. 请求到达服务器，进入等待队列。
        2. GPU完成上一批任务后，调度器查看队列。
        3. 调度器根据预设的最大批次大小，将当前队列中的多个请求（例如N个）组合成一个新批次。
        4. 整个批次的输入Tokens一起通过模型的计算层（如Transformer的Attention和FFN），同时开始，同时结束。
        5. 输出所有结果后，再开始处理下一个动态批次。
      • 优点：
        • 相比静态批处理，提升了GPU利用率和吞吐量。
        • 能更好地应对请求流的波动。
      • 缺点/瓶颈：
        • 依然存在“木桶效应”：一个批次内，如果某个请求需要生成的输出很长（比如1000个token），而其他请求很短（比如10个token），那么生成长文本的请求会拖累整个批次，导致短请求也要等待很久才能返回。

    # 2. 连续批处理

    连续批处理是为了解决动态批处理的“木桶效应”而生的，它是当前大模型推理服务的核心技术。

    • 核心思想：打破“批次”的界限。不再要求一个批次内的请求同时开始、同时结束。它允许已经完成输出的请求提前离开批次，并将新的等待请求加入当前正在运行的批次中，从而实现批次的“连续”或“流动”。

    • 关键机制：Iteration级别的调度。大模型的生成过程是自回归的，每次迭代只生成一个Token。连续批处理在每一次生成Token的迭代中，都会重新决策批次的组成。

    • 工作流程：

      1. 初始时，GPU上有一个批次（Batch）的请求，每个请求都处于“正在生成”状态。
      2. 每一步迭代（生成一个Token）：
        • 生成：当前批次中的所有请求并行地通过模型计算，生成下一个Token。
        • 检查：检查每个请求的状态。
          • 如果某个请求生成了结束符（<eos>），或者达到了最大生成长度，则标记为“完成”。
        • 更新批次：
          • 释放：将已“完成”的请求从当前活动批次中移除，并返回结果给用户。
          • 加入：将等待队列中的新请求加入到当前活动批次中。这些新请求从第一步（迭代）开始计算。
      3. 这个过程在每一步迭代中循环进行，批次成员动态变化。

    • 优点：

      • 彻底解决了“木桶效应”：短请求可以快速完成并返回，不会被长请求阻塞。
      • 极高的GPU利用率：GPU几乎在每个时刻都在处理满负荷的计算，因为新的请求可以立即填补已完成请求留下的空位。
      • 显著降低延迟：尤其是对于短文本交互场景，用户体验大幅提升。

    • 挑战：

      • 实现复杂：需要深度学习框架和推理引擎在底层进行大量优化，以支持动态变化的内存管理和计算调度。
      • KVCache管理：需要高效地分配和释放每个请求的Key-Value缓存。

    # 一个生动的比喻

    • 静态批处理：像一辆固定发车时间和大小的巴士。人满发车，不到站不停。即使有人只坐一站，也要等所有乘客到终点。
    • 动态批处理：像一辆人满就发车的巴士。车的大小可以变化，但还是一辆车把所有乘客同时送到各自目的地。如果有人要去很远的地方，全车人都得等着。
    • 连续批处理：像一列地铁。站点（迭代） 非常密集。每个站点（每生成一个Token）都有人下车（请求完成），也允许新的人上车（新请求加入）。每个人都可以在离自己目的地最近的站点下车，无需等待其他人。

    # 总结

    连续批处理不是对动态批处理的简单否定，而是一次质的飞跃。

    • 它通过将调度粒度从“请求级”细化到“Token级”，实现了计算资源的近乎完美的利用，并大幅降低了用户感知的延迟。

    • 目前，主流的推理框架如 NVIDIA TensorRT-LLM, vLLM, TGI 等，其高性能的核心正是实现了连续批处理技术。

    • 对于需要服务大规模、交互式用户请求的大模型应用来说，连续批处理是必不可少的技术。
